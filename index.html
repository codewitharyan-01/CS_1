Q 01

Define system software. Give difference between System Software and Application software.

▼

Definition — System Software

System software is a collection of programs designed to operate, control, and extend the processing capabilities of a computer. It acts as an interface between the hardware and the application programs. It manages and controls hardware so that application software can perform a task.

Examples: Operating System, Compiler, Assembler, Linker, Loader, Device Drivers.

Q 02

Explain the various stages of the life cycle of a source program with a neat diagram.

▼

Source Program Life Cycle

A source program goes through several transformations before it becomes an executable program. The main stages are:

  1. Source Program – Written by programmer in high-level language (C, Java, etc.)
  2. Preprocessor – Handles macros, file inclusions, conditional compilation (#include, #define)
  3. Compiler – Translates source code into assembly or object code, checks syntax & semantics
  4. Assembler – Converts assembly language code into relocatable machine code (object file)
  5. Linker – Combines multiple object files and library routines into a single relocatable module
  6. Loader – Loads the executable into main memory for execution
  7. Execution – CPU executes the program

Each stage produces an intermediate form of the program which is fed as input to the next stage.

Q 03

Define: System Software, Semantic Gap, Execution Gap, Specification Gap, Token

▼

1. System Software

System Software is a set of programs designed to control and manage the computer hardware and provide services to application programs. It acts as an intermediary between hardware and user applications. Examples: OS, Compiler, Assembler, Loader.

2. Semantic Gap

Semantic Gap is the difference between the meaning/functionality provided by a high-level language and the actual instructions available in the machine language (ISA). It represents the conceptual distance between what a programmer wants to express and what the hardware can directly execute. A larger semantic gap means more complex translation required.

3. Execution Gap

Execution Gap is the difference between the form of a program that a programmer writes (source code) and the form actually executed by the hardware (machine code). It represents the gap that compilers, assemblers, and loaders bridge.

4. Specification Gap

Specification Gap is the difference between the user's requirements/specification and the actual implementation in code. It refers to the mismatch between what the user wants and what the system actually does — often filled by requirements engineering and software design tools.

5. Token

Token is the smallest meaningful unit of a program that the lexical analyzer (scanner) recognizes. It is a sequence of characters that represents a single, logically cohesive unit.

Token types include: Keywords (if, while, int), Identifiers (variable names), Literals (numbers, strings), Operators (+, -, *), Punctuation ({, ;, }).

Source: int a = 10; Tokens: <keyword, int> <id, a> <op, => <literal, 10> <punct, ;>

Q 05

Define: Linker, Loader, Parser, Assembler, Compiler

▼

1. Linker

A Linker is a system program that combines one or more object files (generated by assembler/compiler) along with library files into a single executable file. It resolves external references and assigns final addresses. Types: Static Linker, Dynamic Linker.

2. Loader

A Loader is a system program that loads the executable program into main memory (RAM) for execution. It allocates memory, performs address relocation, and transfers control to the program's entry point. Types: Absolute Loader, Relocating Loader, Direct Linking Loader.

3. Parser

A Parser is a component of the compiler (Phase 2 – Syntax Analysis) that reads a stream of tokens produced by the lexer and checks them against the grammar rules of the language. It builds a parse tree (syntax tree). Types: Top-down Parser, Bottom-up Parser.

4. Assembler

An Assembler is a translator that converts assembly language programs into machine language (object code). It converts mnemonics (ADD, MOV, etc.) to their binary opcode equivalents. Types: Single-pass Assembler, Two-pass Assembler.

5. Compiler

A Compiler is a program that translates the entire source code written in a high-level language into machine code (or assembly language) at once, reporting all errors after complete translation. Output is an executable or object file.

Detailed Explanation: Lexical Analysis

The Lexical Analyzer (also called Scanner) is the first phase of the compiler. It reads the source program character by character and groups them into meaningful sequences called lexemes, then converts them to tokens.

Tasks performed:

  • Remove whitespace and comments
  • Identify tokens (keywords, identifiers, operators, literals)
  • Enter identifiers into the symbol table
  • Report lexical errors (invalid characters)
Source: x = a + b * 2; Tokens produced: <id, x> <op, => <id, a> <op, +> <id, b> <op, *> <num, 2> <punct, ;>

Symbol Table

A Symbol Table is a data structure maintained throughout the compilation process to store information about identifiers (variables, functions, objects) declared in the program.

Information stored in Symbol Table:

  • Name of the identifier
  • Type (int, float, char, etc.)
  • Scope (local, global)
  • Memory address / offset
  • Size
  • For functions: number/type of parameters, return type

Symbol tables are implemented using arrays, linked lists, or hash tables. Hash tables are most efficient with O(1) average-case access.

Q 07

Which are the different assembly language statements? Explain each of them with suitable example.

▼

Types of Assembly Language Statements

An assembly language program consists of 3 types of statements:

1. Imperative Statements (Machine Instructions)

These cause an action to be performed during execution. They correspond directly to machine instructions. Each has an opcode and optional operands.

Syntax: [Label] Mnemonic [Operand1 [, Operand2]] Examples: LOOP MOVER AREG, A ; Move value at A into AREG ADD AREG, B ; Add value of B to AREG MOVEM AREG, C ; Move AREG content to C BC LT, BACK ; Branch if less than

2. Declarative Statements (Assembler Directives / Non-executable)

These provide data or information to the assembler but do not generate any machine code themselves.

DS (Define Storage): A DS 5 ; Reserve 5 words for A DC (Define Constant): B DC '10' ; Define constant 10 at B EQU (Equate): X EQU A+1 ; X is same address as A+1 LTORG: LTORG ; Place literals here

3. Assembler Directives

Instructions for controlling the assembly process.

START 100 ; Start assembly at address 100 END ; End of program ORIGIN 200 ; Set location counter to 200 LTORG ; Generate literal pool at this point
TypeExamplePurpose
ImperativeMOVER AREG, AGenerates machine code; executed at runtime
DeclarativeA DS 3Allocates memory / defines constant

Q 08

Explain advanced assembler directives with suitable example.

▼

Advanced Assembler Directives

These are special instructions that control the assembly process but do not generate machine code.

1. EQU (Equate)

Assigns a name to a constant value or expression.

D EQU A+1 ; D is assigned the address A+1 ; Useful for symbolic constants

2. ORIGIN

Resets the Location Counter (LC) to a specified value.

START 100 A DS 1 ; Address 100 ORIGIN 200 B DS 1 ; Address 200 (jumped from 101)

3. LTORG (Literal Origin)

Forces the assembler to allocate all accumulated literals at the current point in the program.

MOVER AREG, ='5' ADD AREG, ='10' LTORG ; Literal pool placed HERE ; ='5' → 505, ='10' → 506 MOVER BREG, ='15' END ; Second literal pool at end

4. USING / DROP (Base Register Directives)

Tells the assembler which register to use as base register for address calculation.

USING *, R1 ; Use R1 as base from here L R2, VALUE DROP R1 ; Stop using R1 as base

5. MACRO / MEND

Defines a macro (reusable block of code).

MACRO INCR &X MOVER AREG, &X ADD AREG, ='1' MOVEM AREG, &X MEND ; Usage: INCR A ; expands to the 3 instructions above with &X=A
LTORG is important to avoid large offset problems in base-displacement addressing when literals are far from the instruction.

Q 10

Explain forward reference with suitable examples. How to solve it using back-patching?

▼

Forward Reference

A forward reference occurs when a label or symbol is used before it is defined in the program. Since the assembler processes code sequentially, it doesn't know the address of a symbol that hasn't been encountered yet.

Example: MOVER AREG, A ; 'A' is used here at address 100 ADD AREG, ='5' STOP A DS 1 ; 'A' is DEFINED here at address 103 END → 'A' is a forward reference because it is used before it is defined.

Problems Caused

  • In single-pass assembler, address of forward-referenced symbol is unknown at time of use
  • Cannot directly generate correct machine code in one pass

Solution: Back-Patching

Back-patching is the technique used to resolve forward references in a single-pass assembler.

Mechanism:

  1. When a forward reference is encountered, generate incomplete machine code with a placeholder (e.g., 0 or blank) for the unknown address
  2. Add an entry to a Pending List / Incomplete Table recording: the instruction address, and the symbol referenced
  3. When the symbol is defined later, its address is determined
  4. Go back and patch (fill in) all incomplete entries in the generated code with the now-known address
Address Instruction Action 100 MOVER AREG, ? → Add (100, A) to pending list 101 ADD AREG, ='5' 102 STOP 103 DS 1 → A defined at 103 ↑ Patch address 100: MOVER AREG, 103 ✓
Two-pass assemblers solve forward references naturally: Pass-I collects all symbol definitions; Pass-II generates complete code using the built symbol table.

Two Pass Assembler — Pass I

Goal: Build Symbol Table, Literal Table, Pool Table; Generate Intermediate Code.

  1. Initialize Location Counter (LC)
  2. For each statement:
    • If label present → add (label, LC) to SYMTAB
    • If literal present → add to LITTAB
    • If LTORG or END → assign addresses to literals, add entry to POOLTAB
    • If START → set LC
    • Increment LC appropriately
    • Generate intermediate code token-by-token

Two Pass Assembler — Pass II

Goal: Generate final machine code using tables from Pass-I.

  1. Read intermediate code
  2. For each instruction:
    • Look up opcode in OPTAB
    • Resolve symbol references using SYMTAB
    • Resolve literal references using LITTAB
    • Generate binary machine code
Example Source Program: START 100 A DS 3 L1 MOVER AREG, B ADD AREG, ='13' STOP B DC '19' END Pass-I SYMTAB: Pass-I LITTAB: A → 100 ='13' → 105 L1 → 101 B → 105 Pass-II Machine Code (example): 101: 04 1 105 (MOVER AREG, B) 102: 01 1 106 (ADD AREG, ='13' literal at 106) 103: 00 0 000 (STOP)

Q 13

What is grammar? List and explain various types of grammar.

▼

Grammar

A formal grammar G is defined as a 4-tuple: G = (V, T, P, S) where:

  • V = Set of non-terminal symbols (variables)
  • T = Set of terminal symbols
  • P = Set of production rules
  • S = Start symbol (S ∈ V)

Grammar specifies the syntax rules for a language.

Chomsky Hierarchy — 4 Types of Grammar

TypeNameProduction Rule FormAutomaton
Type 0Unrestricted / Phrase Structureα → β (no restriction; α,β ∈ (V∪T)*)Turing Machine
Type 1Context-SensitiveαAβ → αγβ (|LHS| ≤ |RHS|)Linear Bounded Automaton
Type 2Context-Free Grammar (CFG)A → α (single non-terminal on LHS)Pushdown Automaton
Type 3Regular GrammarA → aB or A → a (right-linear)Finite Automaton

Type 2 — Context-Free Grammar (Most important for compilers)

All productions are of form A → α where A is a single non-terminal and α is any string of terminals/non-terminals.

Example CFG for arithmetic expressions: E → E + T | T T → T * F | F F → ( E ) | id

Type 3 — Regular Grammar

Used for lexical analysis (tokenization). Right-linear: A → aB or A → a

Example (identifiers): S → aA | bA A → aA | bA | ε ; accepts (a|b)+
In compiler design: Type 3 is used in Lexical Analysis; Type 2 (CFG) is used in Syntax Analysis (Parsing).
Q 14

Explain Ambiguous Grammar with example.

▼

Ambiguous Grammar

A grammar G is said to be ambiguous if there exists at least one string in the language L(G) that has more than one parse tree (or equivalently, more than one leftmost/rightmost derivation).

Example

Consider grammar: E → E + E | E * E | id

Parse the string: id + id * id

This string has two different parse trees:

Parse Tree 1: (Treats + first — incorrect precedence) E / | \ E + E | / | \ id E * E | | id id Result: (id + id) * id Parse Tree 2: (Treats * first — correct precedence) E / | \ E + E | / | \ id E * E | | id id Structured as: id + (id * id)

Since two parse trees exist, the grammar is ambiguous.

Why Ambiguity is a Problem

  • A compiler cannot decide which parse tree to use, hence which code to generate
  • The meaning of the program becomes unclear

Resolving Ambiguity

  • Operator Precedence — Give * higher precedence than +
  • Associativity rules — Make operators left or right associative
  • Rewrite grammar to be unambiguous:
Unambiguous version: E → E + T | T T → T * F | F F → ( E ) | id ; Now * has higher precedence than + and both are left-associative
Every ambiguous grammar can be rewritten as an unambiguous one, though not every ambiguous language can be described by an unambiguous grammar.

Q 16

What is left recursion? Eliminate left recursion from: S→A, A→aB/Ad, B→bBC/f, C→g

▼

Left Recursion

A grammar is left-recursive if it has a non-terminal A such that there is a derivation A ⟹ Aα for some string α. Left recursion causes top-down parsers (like LL parsers) to loop infinitely.

Direct Left Recursion: A → Aα | β

Elimination Formula: Replace A → Aα | β with:

A → β A' A' → α A' | ε

Given Grammar

S → A A → aB | Ad ← Left recursive (A → Ad) B → bBC | f C → g

Step-by-Step Elimination

Only A → aB | Ad has left recursion. Here:

  • α = d (the recursive part without leading A)
  • β = aB (the non-left-recursive alternative)

Apply the formula:

A → aB A' A' → d A' | ε

Final Grammar (After Left Recursion Elimination)

S → A A → aB A' A' → d A' | ε B → bBC | f C → g
S, B, C have no left recursion, so they remain unchanged. Only A needed modification.

Q 18

Discuss the need for left factoring in grammar? Explain it with suitable example.

▼

Need for Left Factoring

Left factoring is required when a grammar has two or more productions for the same non-terminal that begin with the same terminal symbol. In such cases, a top-down parser (LL parser) cannot determine which production to apply by looking at only the current input token (lookahead).

Left factoring transforms the grammar to delay the decision until enough of the input has been seen to make an unambiguous choice.

Problem Example

Consider: A → αβ₁ | αβ₂ On seeing α, parser cannot decide whether to use αβ₁ or αβ₂ → CONFLICT!

Left Factoring Rule

If A → αβ₁ | αβ₂ | ... | αβₙ | γ then replace with:

A → α A' | γ A' → β₁ | β₂ | ... | βₙ

Practical Example

Original Grammar: stmt → if expr then stmt else stmt | if expr then stmt Both productions start with "if expr then stmt" — conflict! After Left Factoring: stmt → if expr then stmt stmt' stmt' → else stmt | ε Now the parser sees "if", uses the first rule, and defers the else decision to stmt'.

Another Example

Original: A → aAB | aB | b Common prefix: 'a' After Left Factoring: A → aA' | b A' → AB | B

Benefits

  • Removes FIRST/FIRST conflicts from LL(1) parser table
  • Makes grammar suitable for predictive (LL) parsing
  • No change in the language generated
Left factoring does NOT change the language. It only restructures productions for deterministic top-down parsing.
